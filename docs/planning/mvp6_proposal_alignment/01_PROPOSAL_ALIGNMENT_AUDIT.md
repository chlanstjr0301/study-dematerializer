# 01 — Proposal Alignment Audit

## Purpose

Compare the final proposal requirements against the current repository implementation
(post-MVP5). Identify gaps, preserved infrastructure, and what needs modification for MVP6.

---

## Audit Method

Source of truth for proposal requirements:
- `docs/gonghaebun-planning/00_vision.md` through `09_risks_and_guardrails.md`
- `docs/gonghaebun-planning/05_user_session_flow.md` (8-step session)
- `docs/diagnostics/MVP4_PRODUCT_ALIGNMENT_DIAGNOSIS.md` (100-point audit)
- `docs/MVP5_HANDOFF.md` (what MVP5 delivered)
- Task prompt requirements (Ground Truth Cards, Mapping Tasks, Confusion Map)

Current implementation baseline:
- MVP5 complete (6-step integrated session)
- ~975 tests passing
- Mock LLM default, OpenAI provider available
- 3 seed concepts (compactness, connectedness, uniform_continuity)

## Where Current System Behaves Like a Question-Bank Product

The MVP4 diagnosis (`docs/diagnostics/MVP4_PRODUCT_ALIGNMENT_DIAGNOSIS.md`) identified this.
Post-MVP5, the following question-bank behaviors remain:

1. **BankBrowser, BankReview pages**: These are developer tools for managing generated questions.
   They serve the question-bank mental model, not the learner flow.
   **Action**: Keep under "개발자 도구" toggle (already done). Do not expand.

2. **RecallSession standalone page**: Allows learner to do recall without going through the full
   study session. This is useful for review but can become a flashcard pattern.
   **Action**: Keep for review-due use case. But primary learning path must be StudySession.

3. **GET /api/bank/{id}**: Returns raw question list. Useful for dev, not for learner.
   **Action**: Keep as-is. New mapping task API is the learner-facing equivalent.

4. **Question bank auto-generation from pipeline**: Stage 6 generates questions and saves to banks.
   These questions are representation-typed but not card-grounded.
   **Action**: In MVP6, mapping tasks replace free question generation as the primary
   assessment mechanism. Keep question bank for backward compat but de-emphasize.

## Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Ground Truth Card quality determines everything downstream | High | Start with compactness only. Validate against Rudin. Have clear required_terms. |
| 7-step session makes UX longer | Medium | Mapping step is 3 tasks, not open-ended. Time-box to ~5 min. |
| Confusion Map adds complexity without proven learning benefit | Medium | Start with display-only. Don't gate progression on confusion map. |
| Deterministic evaluator may be too rigid | Medium | Use required_terms overlap + known misconception matching. Fall back to needs_human_review for ambiguous cases. |
| Modifying study_session_service.py is risky (central file) | Medium | Add mapping/confusion as separate service modules. Minimize changes to existing step handlers. |
